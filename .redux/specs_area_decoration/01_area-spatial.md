## Summary
This section contains pure data-structure plumbing with no RNG: (1) PreviewState, a non-destructive preview layer with a per-exact-position hash index (16-bit packed x/y/z key) plus componentwise min/max bounds; (2) AreaDefinition, which expands an area into a Position list three ways — inclusive triple-loop rectangle (z->y->x, normalized per axis), 4-connected BFS flood fill bounded by inclusive MANHATTAN distance (diamond) on a single z-floor, or a verbatim copy of the live editor selection (every selected tile, no filtering, vector insertion order); (3) SpatialHashGrid, an 8-cell-default 2D (z-ignored) hash grid whose queryRadius scans a padded square of cells then exact-filters by inclusive CHEBYSHEV distance; (4) PreviewManager, a trivial singleton holding a raw PreviewState pointer plus a 0.7 preview opacity. All operate on Position{int x,y,z} (default (0,0,0)).

## Semantics
All coordinates are C++ `int` (Position { int x, y, z; default ctor = (0,0,0) }, defined in b:/Github/rme_redux/source/map/position.h).

=== PreviewState (header fields: items: vector<PreviewItem{position, itemId:uint16 default 0, sourceRule: FloorRule* default null}>, totalItemsPlaced:int=0, itemCountById: map<uint16,int>, placementsByRule: map<FloorRule*,int>, seed:uint64=0, isValid:bool=false, errorMessage:string, minPos/maxPos: Position; private spatialIndex: map<uint64, vector<size_t>>) ===

PreviewState::clear() — resets ALL of, in this order: items -> empty; totalItemsPlaced = 0; itemCountById -> empty; placementsByRule -> empty; seed = 0; isValid = false; errorMessage -> ""; spatialIndex -> empty; minPos = Position(0,0,0); maxPos = Position(0,0,0).

PreviewState::positionHash(pos) [static, private] — key = ((x & 0xFFFF) << 32) | ((y & 0xFFFF) << 16) | (z & 0xFFFF), all as unsigned 64-bit. Each coordinate is masked to its LOW 16 BITS first (so coordinates that differ by a multiple of 65536 collide; negative coords wrap two's-complement, e.g. -1 -> 0xFFFF). Layout: x in bits 32..47, y in bits 16..31, z in bits 0..15; no field overlap.

PreviewState::hasItemAt(pos) — h = positionHash(pos); returns true iff spatialIndex contains key h AND that bucket vector is non-empty. (In practice buckets are never empty after rebuild, but the emptiness check exists.)

PreviewState::getItemsAt(pos) — result = empty list; h = positionHash(pos); if bucket exists, for each stored index idx IN BUCKET ORDER: if idx < items.size() (defensive bounds guard — stale indices silently skipped), append pointer/reference to items[idx]. Returns list. Bucket order = ascending item index (see rebuild). Does NOT verify items[idx].position == pos, so hash collisions (coords mod 65536) would return wrong-position items.

PreviewState::rebuildSpatialIndex() — Step 1: spatialIndex.clear(). Step 2: for i = 0 .. items.size()-1 in order: append i to spatialIndex[positionHash(items[i].position)] (so each bucket holds ascending indices). Step 3: ONLY IF items is non-empty: minPos = maxPos = items[0].position; then for EVERY item (including items[0] again, harmless): minPos.x = min(minPos.x, it.x); minPos.y = min; minPos.z = min; maxPos.x = max; maxPos.y = max; maxPos.z = max (componentwise, independent axes). If items IS empty, minPos/maxPos are left untouched (NOT reset) — only clear() resets them to (0,0,0).

=== AreaDefinition (fields: type in {Rectangle, FloodFill, Selection}, default Rectangle; rectMin/rectMax: Position; floodOrigin: Position; floodTargetGround: uint16 = 0; floodMaxRadius: int = 100) ===

getAllPositions(map) — dispatch on type: Rectangle -> getPositionsRectangle() (map unused); FloodFill -> getPositionsFloodFill(map); Selection -> getPositionsFromSelection(map). Falls through to empty list for an out-of-enum value.

contains(pos) — ONLY implemented for Rectangle: returns pos.x >= rectMin.x && pos.x <= rectMax.x && pos.y >= rectMin.y && pos.y <= rectMax.y && pos.z >= rectMin.z && pos.z <= rectMax.z (inclusive on all 6 bounds, using rectMin/rectMax RAW — NOT normalized, so an inverted rect where rectMin > rectMax makes contains() false for every position). For FloodFill and Selection types it ALWAYS returns false (default case).

getBounds(outMin, outMax) — Rectangle: outMin = rectMin, outMax = rectMax verbatim (again NOT normalized). FloodFill: outMin = (floodOrigin.x - floodMaxRadius, floodOrigin.y - floodMaxRadius, floodOrigin.z); outMax = (floodOrigin.x + floodMaxRadius, floodOrigin.y + floodMaxRadius, floodOrigin.z) — i.e. a SQUARE bounding box (Chebyshev box) around the origin even though the flood itself is bounded by Manhattan distance, so bounds are a strict superset of reachable tiles; z is fixed to origin z. Selection (and any other): outMin = outMax = (0,0,0).

getPositionsRectangle() [private] — normalizes EACH axis independently: minX = min(rectMin.x, rectMax.x), maxX = max(...), same for Y and Z (so inverted rects DO work here, unlike contains/getBounds). Reserves (maxX-minX+1)*(maxY-minY+1)*(maxZ-minZ+1) entries. Triple nested loop, ALL bounds inclusive (<=): outer z from minZ to maxZ, middle y from minY to maxY, inner x from minX to maxX; appends Position(x,y,z). Output order: x fastest, then y, then z.

getPositionsFloodFill(map) [private] — BFS with FIFO queue and a visited set keyed by the SAME 16-bit packed hash formula as PreviewState::positionHash ((x&0xFFFF)<<32 | (y&0xFFFF)<<16 | (z&0xFFFF)).
  1. targetGround = floodTargetGround. If targetGround == 0: originTile = map.getTile(floodOrigin); if originTile is null OR originTile->ground is null -> RETURN EMPTY immediately. Else targetGround = originTile->ground->getID() (the ground item's ID).
  2. Enqueue floodOrigin.
  3. While queue non-empty: pop FRONT pos, then apply these checks IN THIS EXACT ORDER, each `continue`s (skips to next queue entry) on failure:
     a. dist = abs(pos.x - floodOrigin.x) + abs(pos.y - floodOrigin.y)  [MANHATTAN / L1, z ignored]. If dist > floodMaxRadius -> continue. So radius is INCLUSIVE: tiles at exactly dist == floodMaxRadius are accepted AND their neighbors are enqueued (those neighbors then fail this check). Area shape = diamond. NOTE: this check runs BEFORE the visited check, so out-of-radius positions are never marked visited and can be re-enqueued repeatedly by multiple in-radius neighbors (pure perf cost, no behavior change).
     b. hash pos; if hash already in visited -> continue; else INSERT into visited NOW (before tile checks — so non-matching tiles also become visited and are never re-examined).
     c. tile = map.getTile(pos); if tile null OR tile->ground null -> continue.
     d. if tile->ground->getID() != targetGround -> continue.
     e. Append pos to result.
     f. Enqueue 4 ORTHOGONAL neighbors in this exact order: (x-1, y, z) WEST, (x+1, y, z) EAST, (x, y-1, z) NORTH, (x, y+1, z) SOUTH. 4-CONNECTIVITY ONLY — NO diagonals. z NEVER changes (single-floor flood).
  4. Return result. Result order is BFS order of tiles that passed all checks; first element is floodOrigin itself (when it matches). KEY: neighbors are only enqueued from MATCHING tiles (step f is after step e), so if the origin tile fails (null/no ground/wrong ground when floodTargetGround != 0), the result is empty — the flood cannot "jump over" non-matching tiles.

getPositionsFromSelection(map) [private] — The `map` parameter is IGNORED. editor = g_gui.GetCurrentEditor(); if null -> return empty. tiles = editor->selection.getTiles(), which returns `const std::vector<Tile*>&` (selection insertion order — deterministic vector, see source/editor/selection.h:112-113,138). For EVERY tile pointer in that vector (no null check, no ground check, no blocking check, no z filter): append tile->getPosition(). Includes ALL selected tiles unconditionally, on every floor, with or without ground, duplicates impossible only insofar as the selection vector itself has no duplicates.

=== SpatialHashGrid (m_cellSize: int, ctor default 8; m_cells: unordered_map<uint64, vector<pair<Position,size_t>>>) ===

ctor(cellSize = 8) — stores cellSize verbatim (no validation; 0 would divide-by-zero).

insert(pos, itemIndex) — cx = pos.x / m_cellSize; cy = pos.y / m_cellSize  [C++ INTEGER DIVISION = TRUNCATION TOWARD ZERO, not floor — matters for negative coords]. cellHash = (uint64(cx) << 32) | uint64(cy). NOTE: cx/cy are NOT masked; a negative cy sign-extends to all-ones in 64 bits and its high 32 bits OR over the cx field (see edgeCases). z is NOT part of the key — the grid is purely 2D; items on different floors share cells. Appends pair (pos, itemIndex) to the cell bucket (insertion order preserved; duplicates allowed).

queryRadius(center, radius) — result = empty. cellRadius = (radius / m_cellSize) + 1 (integer truncating division, then +1 padding — guarantees coverage). cx = center.x / m_cellSize; cy = center.y / m_cellSize (truncating). Double loop: OUTER dy from -cellRadius to +cellRadius inclusive, INNER dx from -cellRadius to +cellRadius inclusive; compute cellHash = (uint64(cx+dx) << 32) | uint64(cy+dy); if that cell exists, for each stored (pos, idx) entry IN BUCKET INSERTION ORDER: dist = max(abs(pos.x - center.x), abs(pos.y - center.y))  [CHEBYSHEV / L-infinity, z IGNORED]; if dist <= radius (INCLUSIVE) append idx to result. Returns indices; each inserted item appears at most once per insertion (it lives in exactly one cell). So: bounding cells scan THEN exact Chebyshev filter; matches form a (2*radius+1)^2 square around center across ALL z floors. Result order: row of cells top-to-bottom (dy), left-to-right (dx), then per-cell insertion order. radius 0 returns items at exactly (center.x, center.y) on any z. Negative radius: cellRadius = (neg/cs)+1 may still be >= 0 (e.g. radius=-1, cs=8 -> cellRadius = 0+1 = 1, cells scanned) but the dist <= radius filter rejects everything, so result is empty.

clear() — m_cells.clear() only (m_cellSize retained).

## Constants
- floodMaxRadius (default) = 100 -- AreaDefinition default flood-fill cap; inclusive MANHATTAN-distance bound from floodOrigin (diamond-shaped limit), z ignored
- floodTargetGround (default) = 0 -- 0 = sentinel meaning 'derive target ground ID from the tile at floodOrigin'; if origin tile/ground missing, flood returns empty
- SpatialHashGrid cellSize (default) = 8 -- Side length of square 2D hash cells; cell coords via truncating integer division x/8, y/8; z not used
- queryRadius cell padding = +1 -- cellRadius = (radius / cellSize) + 1; scan window is (2*cellRadius+1)^2 cells before exact filtering
- position hash mask = 0xFFFF -- Each of x, y, z masked to low 16 bits before packing into key ((x&0xFFFF)<<32 | (y&0xFFFF)<<16 | (z&0xFFFF)) — used identically by PreviewState::positionHash and the flood-fill visited set
- queryRadius metric = Chebyshev (L-infinity), inclusive -- dist = max(|dx|,|dy|) <= radius; z ignored; matches a (2r+1)x(2r+1) square
- flood-fill metric = Manhattan (L1), inclusive -- dist = |dx|+|dy| <= floodMaxRadius; checked before visited-set check
- flood-fill connectivity = 4 (orthogonal) -- Neighbors enqueued in order W(x-1), E(x+1), N(y-1), S(y+1); same z; NO diagonals
- m_previewOpacity (default) = 0.7f -- PreviewManager default render opacity for preview items; setter unclamped
- Position default = (0,0,0) -- Position() zero-initializes; used by clear() for minPos/maxPos and by getBounds() Selection fallback

## Edge cases
1) Inverted rectangle (rectMin > rectMax on any axis): getPositionsRectangle NORMALIZES per axis and still yields the full box, but contains() and getBounds() use the raw fields — contains() returns false for everything and getBounds() returns inverted bounds. Inconsistent by design; port as-is. 2) contains() returns false for FloodFill and Selection area types unconditionally. 3) Flood fill with floodTargetGround==0 and missing origin tile or origin tile without ground -> empty result. With explicit floodTargetGround != 0 and a non-matching origin tile -> also empty (neighbors only enqueued from matching tiles; flood cannot cross gaps). 4) Flood-fill tiles at exactly dist == floodMaxRadius are included; out-of-radius positions are skipped BEFORE being marked visited, so they can be re-enqueued many times (perf only, same output). 5) Position hash masks coords to 16 bits: positions differing by 65536 on an axis collide in both PreviewState.spatialIndex and the flood visited set (PreviewState::getItemsAt would return wrong-position items; flood would falsely treat a tile as visited). Irrelevant for normal map sizes (<= 65535) but real for negative/huge coords. 6) SpatialHashGrid key ((uint64)cx << 32) | (uint64)cy: a NEGATIVE cy sign-extends to 64 bits and its upper 32 bits OR over the cx field — e.g. (cx=0, cy=-1) and (cx=-1, cy=-1) both hash to 0xFFFFFFFFFFFFFFFF. Cell-key collisions for negative coordinates; harmless on maps with non-negative coords (collisions only ADD candidates which the exact Chebyshev filter then rejects... but they can also MERGE distinct cells, still corrected by the exact filter in queryRadius — net effect: correct results, only when the merged cell happens to be scanned; for insert+query consistency the same wrong key is computed on both sides, so behavior stays self-consistent). 7) SpatialHashGrid ignores z entirely (insert key and queryRadius distance) — items on different floors block/satisfy spacing checks against each other. 8) queryRadius with radius 0 returns items sharing center's x,y on any z; negative radius returns empty (filter rejects all). 9) rebuildSpatialIndex on an empty items list leaves minPos/maxPos UNCHANGED (stale) — only clear() resets them to (0,0,0). 10) getItemsAt skips indices >= items.size() (only possible if index is stale relative to items). 11) getPositionsFromSelection ignores its map parameter and reads the LIVE current editor's selection; returns empty if no current editor; includes every selected tile with no filtering whatsoever. 12) cellSize is not validated; 0 would be a division by zero (default ctor always passes 8). 13) hasActivePreview requires isValid==true, so a preview that was set but later invalidated yields empty getPreviewItemsAt without being detached.

## Porting notes
INDEXING: items is a 0-based C++ vector and spatialIndex stores 0-based indices; in Lua use 1-based tables but keep RELATIVE order (ascending). getItemsAt's `idx < items.size()` guard becomes `idx <= #items` for 1-based. INTEGER DIVISION: C++ `/` on ints TRUNCATES TOWARD ZERO; Lua `//` and math.floor FLOOR toward negative infinity — these differ for negative operands. SpatialHashGrid (pos.x / cellSize, pos.y / cellSize, radius / cellSize) must use truncation: e.g. `local q = x >= 0 and x // cs or -((-x) // cs)`, or math.tointeger-safe trunc. With floor division, negative coordinates land in different cells than C++ and cell-key collisions change — but since insert and query use the same function, results stay correct as long as you are CONSISTENT and keep the exact Chebyshev filter; for byte-faithful candidate ORDER, use truncation. BIT OPS / HASH KEYS: positionHash packs into 48 bits — fits Lua 5.3+ integers exactly; replicate `((x & 0xFFFF) << 32) | ((y & 0xFFFF) << 16) | (z & 0xFFFF)`; Lua's `&` on negative integers is two's-complement like C++, so `-1 & 0xFFFF == 0xFFFF` matches. For the SpatialHashGrid cell key, replicating `((cx % 2^64) << 32) | (cy % 2^64)` including the negative-cy sign-extension collision requires 64-bit math: in Lua 5.3+ `(cx << 32) | cy` on integers reproduces it exactly (Lua ints are 64-bit two's-complement); do NOT 'fix' it by masking cy, or negative-coordinate behavior diverges (flag: this looks like a latent bug — negative cy pollutes the cx field — port as-is). Alternatively use a string key 'cx:cy' but then negative-cell collision behavior intentionally differs; document the choice. CONTAINER ORDER: spatialIndex and m_cells are unordered_maps but the code NEVER iterates them (only point lookups), so Lua table key order is irrelevant here — what MUST be preserved is: bucket append order, queryRadius dy-outer/dx-inner scan order, flood BFS FIFO with W/E/N/S neighbor order, rectangle z/y/x loop nesting, and selection vector order. QUEUE: use a table with head/tail indices for the BFS FIFO (table.remove(t,1) is O(n) but behavior-equivalent). FLOAT: m_previewOpacity 0.7f is a float (0.699999988...); in Lua it becomes double 0.7 — only matters if compared exactly. BUG FLAGS: (a) contains() always false for FloodFill/Selection types — callers relying on it for those types get no containment; (b) contains/getBounds do not normalize inverted rectangles while getPositionsRectangle does; (c) getBounds for FloodFill returns a square (Chebyshev) box around a Manhattan-bounded (diamond) flood — superset, not tight; (d) SpatialHashGrid drops z from both key and distance — cross-floor interference is the as-is behavior, keep it; (e) negative-cy cell-key sign-extension collision described above; (f) rebuildSpatialIndex leaves stale minPos/maxPos when items is empty. Port all six as-is and flag in comments.
