## Summary
The three distribution generators share an identical per-rule placement pipeline (maxItemsTotal cap, per-rule maxPlacements, density gate with friend bias, weighted item pick, cluster-center Chebyshev spacing, build/lookahead-cap/spacing/commit). ## RNG
All draws come from a single shared std::mt19937 m_rng seeded once per preview with the low 32 bits of the seed. Order is strictly sequential through the generator; any conditional skip shifts the stream for everything after it, so exact per-seed reproduction requires matching every conditional draw below.

generatePureRandom draw order:
 1. std::shuffle(indices, m_rng) — Fisher-Yates over tiles.size() elements. The C++ standard leaves the number and mapping of raw engine calls unspecified (MSVC and libstdc++ differ); the permutation itself is what matters semantically.
 2. Then, visiting tiles in shuffled order, per matching rule that survives the global cap (step 1) and maxPlacements (step 2) checks: one uniform_real<float>[0,1) draw (density gate; distribution object constructed fresh per iteration — stateless, no effect on stream).
 3. If the gate passes AND rule.items non-empty AND totalWeight > 0: one uniform_int<int>[0, totalWeight-1] draw (weighted item pick). (No draw at all if items empty or totalWeight <= 0.)
 4. If the picked entry is a cluster ItemEntry AND checkClusterCenterSpacing passed: inside buildPlacementItems, for each additional center i = 1..max(1,clusterCount)-1: up to 20 attempts, each attempt exactly two uniform_int<int>[-max(0,clusterRadius), +max(0,clusterRadius)] draws (dx then dy); attempts end early on first accepted candidate; rejected attempts still consumed their two draws. These draws occur BEFORE buildPlacementItems can fail on tile validation, so map content affects the stream.

generateClustered draw order:
 1. numClusters = min(clusterCount, #tiles) draws of uniform_int<size_t>[0, tiles.size()-1] (center picks, with replacement; zero draws if clusterCount <= 0).
 2. No shuffle. Tiles visited in deterministic sorted order (distance asc, index asc tiebreak). Per rule: same conditional sequence as PureRandom steps 2-4 (density gate draw, conditional item-pick draw, conditional cluster-offset draws).

generateGridBased draw order:
 1. Per grid point in column-major order (gx outer from minPos.x step gridSpacingX; gy inner from minPos.y step gridSpacingY): one uniform_int<int>[-gridJitter, +gridJitter] draw for px, then one for py — ALWAYS consumed per grid point (drawn before the tileMap lookup, so missed probes still advance the stream by two draws... subject to the caveat that uniform_int_distribution's raw engine consumption is implementation-defined; semantically it is one value per call). The two distribution objects are constructed once before the loops.
 2. On a tileMap hit: same conditional per-rule sequence (density gate draw, conditional item-pick draw, conditional cluster-offset draws).

Determinism notes: given the same seed, same preset, same tile list (content and ORDER), and same map occupancy, every generator is fully deterministic. The tile input order matters everywhere: it seeds the shuffle permutation in PureRandom, the index tiebreak and center indexing in Clustered, and tiles[0] (z floor + min/max init) plus tileMap overwrite order in GridBased.

## Edge cases
- tiles empty: Clustered and GridBased return immediately (no RNG). PureRandom shuffles an empty vector (no draws) and the loop body never runs.
- maxItemsTotal == 0: the first top-of-loop cap check trips immediately -> m_previewWasCapped = true, zero placements (PureRandom still consumed the shuffle draws first).
- maxItemsTotal < 0 (default -1): all three cap checks disabled (guarded by >= 0).
- The lookahead cap (totalItemsPlaced + placementSize > maxItemsTotal) uses strict >, allows exactly filling the cap, does NOT set m_previewWasCapped, and only skips the current placement — generation continues and smaller placements may still land.
- Two different "stop" styles when already at cap: PureRandom/Clustered outer-loop check does `break`, the inner per-rule check does `return`; GridBased uses `return` in both spots. Net behavior identical (nothing executes after the loops).
- rule.maxPlacements == 0: rule can never place (0 >= 0 at first check). rulePlacements counts commits per generator call, not items, and operator[] default-inserts 0.
- selectItemFromRule with all weights 0 (or summing <= 0): returns nullptr WITHOUT consuming a draw — shifts the RNG stream relative to a rule with positive weights.
- adjustedDensity boundary: gate is `draw > density -> skip`, so a draw exactly equal to density passes; density >= 1.0 always passes (uniform_real is [0,1)); density 0.0 passes only on an exact 0.0 draw (possible, astronomically rare); negative density never passes.
- Clustered with clusterCount <= 0: zero centers, every tile's distanceScore = FLT_MAX, falloff = exp(-huge) underflows to 0.0f -> density ~0 for every tile (placements essentially impossible).
- Clustered duplicate centers (with-replacement sampling) are harmless: min-distance over duplicates is unchanged.
- Clustered distance ignores z entirely; cluster-entry center spacing (checkClusterCenterSpacing) compares same-z only with Chebyshev distance.
- GridBased multi-floor area: pz is locked to tiles[0].z, so tiles on other floors can never be probed (see portingNotes).
- GridBased jitter can land two grid probes on the same tile (double opportunity) or push probes outside the bounding box / off-area (silent miss after consuming 2 draws).
- GridBased gridJitter == 0: dist(0,0) always yields 0 (px=gx, py=gy) — still a per-point "draw" call semantically.
- Same tile can receive placements from MULTIPLE matching rules in one visit (no break after success); each is independently gated and spacing-checked against already-committed items.
- m_clusterCenters persists across the distribution pass into the later cluster-rule passes within the same preview, so cluster ItemEntry anchors placed here constrain (and are constrained by) nothing retroactively but do constrain later checks.
- buildPlacementItems for cluster entries can return fewer than clusterCount sub-centers (20-attempt exhaustion) or fail entirely after consuming RNG draws.

## Porting notes
- RNG fidelity: std::shuffle, uniform_int_distribution, and uniform_real_distribution are all implementation-defined in how they consume mt19937 outputs; exact per-seed sequences are NOT reproducible across compilers, let alone in Lua. For the port, implement mt19937 (seeded with seed % 2^32) plus your own fixed Fisher-Yates (e.g., for i = n,2,-1: j = draw[1..i]; swap) and fixed int/float mappings, and document that seeds produce different (but internally deterministic) layouts than the C++ build.
- 0- vs 1-based: C++ indices/rolls are 0-based. Weighted pick draws roll in [0, totalWeight-1] with acceptance `roll < cumulative`; in Lua with roll = random(1, totalWeight) use `roll <= cumulative`. Clustered center pick is index in [0, n-1] -> Lua [1, n].
- Density gate is strict `draw > density -> skip` on a [0,1) draw; preserve the <=-passes semantics (Lua math.random() returns [0,1), matching).
- Float vs double: C++ uses single-precision expf/powf/sqrtf and float accumulation for falloff/proximity/density; Lua doubles will diverge at gate boundaries by tiny amounts — acceptable, but do not "optimize" formula order: falloff = exp(((-dist) * clusterStrength) * 0.1), bias = base*(1-fc) + (base*proximity)*fc.
- INCONSISTENCY (behavior as-is, possibly intentional): Clustered applies applyFriendBias AFTER multiplying the falloff (bias operates on the falloff-reduced density), while PureRandom and GridBased apply it to raw rule.density. Port as-is.
- BUG (as-is): generateGridBased never updates minPos.z/maxPos.z, and probes use pz = tiles[0].z. In a multi-floor area, only tiles sharing the FIRST input tile's z can ever be decorated by grid mode. Port the behavior faithfully (or flag for fix), and note it depends on tile input order.
- BUG-ADJACENT (as-is): GridBased has no guard for gridSpacingX/Y <= 0 (infinite loop in C++) and gridJitter < 0 (uniform_int with lo > hi is UB). A Lua port should clamp spacing to >= 1 and jitter to >= 0 — or reproduce the hang if byte-faithfulness is truly required (not recommended).
- Position hash masks coordinates to 16 bits (x<<32 | y<<16 | z); fine for map coords < 65536. In Lua 5.1/LuaJIT without 64-bit integers, use a string key "x:y:z" or two-level tables instead — the numeric hash exceeds 2^48 and loses no precision in doubles (max ~2^48), so a Lua number key also works in 5.3+.
- Container order: rulePlacements and itemCountById/placementsByRule are unordered_maps used ONLY for keyed lookup/increment (never iterated) — Lua tables are a drop-in. matchingRules order comes from floorRules vector order, which was sorted priority-DESC with non-stable std::sort: equal-priority rules have unspecified relative order in C++. Port with a stable sort + original-index tiebreak and document possible divergence for equal priorities.
- Tile input order is semantically load-bearing in all three generators (shuffle permutation, sort tiebreaks, tiles[0] for GridBased bounds/floor, tileMap overwrite). The Lua port must preserve the exact tile collection order of collectTileData (row/column order — covered by another section).
- The per-rule cap `rulePlacements` counts placements (commits), not items; one cluster commit with 12 items counts as 1.
- m_previewWasCapped is set ONLY when totalItemsPlaced is already >= cap at a loop-top/rule-top check; the size-lookahead skip never sets it.
- Remember the rule loop does not break on success — keep the multi-rule-per-tile behavior.
- integer division: none in this section; all distance math is float (Euclidean) or integer Chebyshev (max of abs) — no rounding traps, but keep Chebyshev as integer math.
