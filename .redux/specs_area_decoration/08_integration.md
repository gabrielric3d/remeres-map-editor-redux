# Area Decoration dialogs: default seeding of DecorationPreset/FloorRule, Rule-from-Selection derivation, and Preview/Apply/Reroll user flow (b:/Github/rme_redux/source/ui/dialogs/area_decoration_dialog.cpp, b:/Github/rme_redux/source/ui/dialogs/area_decoration_rule_from_selection_dialog.cpp, b:/Github/rme_redux/source/editor/area_decoration.h)

## Summary
The main AreaDecorationDialog seeds its UI controls with values that exactly mirror the C++ struct defaults (SpacingConfig{1,2,true}, DistributionConfig{PureRandom,0.5,3,3,3,1}, maxItemsTotal=-1, skipBlockedTiles=true, seed unchecked/"0", Rectangle area with Z=7). A fresh FloorRule from "Add Rule" is the struct-default rule with name="New Rule", density=1.0, SingleFloor mode; it is pushed into preset.floorRules BEFORE editing and erased/restored on cancel. "Rule from Selection" builds a Cluster-mode FloorRule: items are bucketed into 4 z-order layers (ground/border/alwaysOnBottom/regular), layers with role "Floor Cluster" (default: ground+borders) become clusterTiles grouped by (x,y) offset relative to the truncated-average selection center (z dropped), layers with role "Items List" (default: bottom+regular) become weighted ItemEntries with weight = occurrenceCount*100. Preview flow: "Apply Changes" builds preset from UI and calls generatePreview(seed) (seed 0 unless "Use Specific Seed"); "Reroll" regenerates ONLY rebuilding the preset if no valid preview exists; "Apply to Map" commits the preview as an undoable ACTION_DRAW batch, auto-looping up to 20 batches while the engine reports the preview was capped; "Reroll and Apply" first removes the last apply, then rerolls and chains into Apply. Preview overlay opacity lives in PreviewManager (default 0.7f) but no code ever calls set/getPreviewOpacity or getPreviewItemsAt outside the engine itself.

4. Per tile (selection iteration order, an unordered set in practice): relOffset = (tilePos.x-center.x, tilePos.y-center.y, tilePos.z-center.z).
   a. If tile->ground && ground->getID()>0: append SelectionItemInfo(groundId, tilePos, relOffset, layer=0).

1. rule = FloorRule defaults; rule.ruleMode=Cluster; rule.name=name text (untrimmed here); rule.instanceCount=spin; rule.instanceMinDistance=minDistanceSpin; rule.density=(float)densitySpin/100.0f. 2. Cluster tiles: ordered std::map keyed by pair(relOffset.x, relOffset.y) — Z IS DISCARDED. Result: per-tile itemIds ordered ground-first, then borders, then (if reassigned to cluster role) bottom/regular — i.e., correct draw stacking when grounds precede borders.
3. Convert clusterMap to rule.clusterTiles iterating the std::map in ASCENDING key order (lexicographic: x first, then y); CompositeTile{offset=Position(x,y,0), itemIds=collected vector}.
4. Then iterate itemCounts in ASCENDING itemId order pushing ItemEntry(itemId, count*100) — weight = total occurrences across role-1 panels, times 100.

=== E. PREVIEW -> APPLY -> REROLL FLOW ===
OnPreview ("Apply Changes" button):
1. No engine -> error box. 2. m_preset.floorRules empty -> error "Add at least one floor rule first". 3. Engine applyPreview (area_decoration.cpp:1924): requires valid AND non-empty preview ("No valid preview to apply"); groups items by 48-bit position hash ((x&0xFFFF)<<32 | (y&0xFFFF)<<16 | (z&0xFFFF)); per position: map.getTile(pos), SKIP silently if tile missing; deep-copy tile; addItem(Item::Create(id)) for every grouped preview item (order = preview items order within that position bucket); register Change. ## RNG
The two dialog files perform ZERO RNG draws themselves. All randomness is delegated to DecorationEngine. Relevant to the flow: (1) generatePreview(seed): if the passed seed is 0, it uses preset.defaultSeed when nonzero, otherwise draws ONE value from std::random_device (rd(); a uniform 32-bit unsigned int) to become the seed. (2) The mt19937 engine is then seeded with static_cast<unsigned int>(seed) — only the LOW 32 BITS of the (possibly 64-bit, user-typed) seed affect generation, while previewState.seed stores and displays the full 64-bit value. For determinism per seed in the Lua port, seed your mt19937-equivalent with seed % 2^32. (3) OnReroll and OnRerollApply call generatePreview(0): with 'Use Specific Seed' UNchecked (and engine preset rebuilt that way) this draws a fresh random_device seed; the resulting seed is written back into the seed text box without checking the checkbox. (4) OnApply's auto-batch loop calls generatePreview(0) between batches — each capped batch gets a new random seed UNLESS preset.defaultSeed != 0, in which case every batch reuses the same seed (see edge cases). (5) DecorationEngine::rerollPreview() (random_device -> generatePreview) exists but the dialog never calls it; the dialog's Reroll uses generatePreview(0) instead. All sequencing of in-engine draws (per-tile density rolls, weighted item selection, jitter, etc.) is in the engine generators and is covered by the engine-section spec; from the dialog side the only ordering guarantee needed is: one engine-level seed decision per generatePreview call, then deterministic engine generation from that seed.

## Edge cases
1. Reroll with a valid existing preview does NOT re-read the UI: changes to spacing/distribution/rules after the last 'Apply Changes' are ignored by Reroll until the preview is invalidated (apply or clear). 2. Reroll/auto-batch with 'Use Specific Seed' checked: generatePreview(0) resolves to preset.defaultSeed, so 'Reroll' regenerates the IDENTICAL preview, and OnApply's capped-batch loop re-applies the same layout up to 20 times (spacing checks run against a cleared spatial hash each generation, so duplicates can stack). Behavior as-is. 3. OnApply counts batchCount BEFORE applyPreview; applyPreview silently skips items whose tile doesn't exist, so 'items applied' stats can overcount; those skipped items are still recorded in m_lastAppliedItems and removeLastApplied will simply not find them (and if NOTHING was found at all it errors but keeps the list). 4. applyPreview with a valid-but-empty preview returns false ('No valid preview to apply') — OnApply then shows an error box even though nothing is wrong semantically. 5. stoull on the seed text throws on non-numeric/negative input -> caught -> seed 0 (random). Seeds > 2^64-1 also throw -> 0. Seeds in (2^32, 2^64) silently truncate to low 32 bits for generation. 6. OnAddRule pushes the rule before editing; cancel removes by captured index — if the user deletes other rules from the main dialog while the non-modal editor is open, the index can point at a different rule (bounds-checked but not identity-checked); same staleness applies to Edit's backup-restore. Behavior as-is. 7. Rule-from-selection: BuildRule uses the UNtrimmed name while OnOK validates the right-trimmed name — a name of only spaces is rejected, but 'name ' is stored with trailing space. 8. Rule-from-selection with a multi-floor (multi-z) selection: cluster grouping keys on (x,y) only, so items from different z floors merge into one CompositeTile at offset z=0. 9. Selection center uses integer division of (min+max); for a 2-tile-wide span the center is biased toward min, so offsets are asymmetric (e.g. -0..+1). 10. ZOrderLayerPanel role radios: GetRole() returns 1 if the floor radio is null OR unchecked. 11. OnAddItem and OnAddItemRange skip the duplicate check that AddItemById performs, so duplicate single-item entries are possible via those buttons. 12. Add Range does not verify item ids exist in the item database — invalid ids become entries with '?' icons. 13. The density spin floors at 1, but LoadRuleData's (int)(density*100) truncation means float densities like 0.4999 load as 49; a load/save round-trip can drift density down by 1% per cycle due to float representation. 14. Dialog close hides (Veto) instead of destroying; the preview is cleared on close but rules/settings persist for the session. 15. UpdateEngine() recreates the engine when the current editor changes — last-applied history and preview are lost with the old engine instance. 16. Friend-floor range validation only triggers when BOTH from and to are >0; from>0,to=0 saves silently and isFriendRange()==false, so friendFloorId=0 means no friend bias despite a partial range being entered.

## Porting notes
1. INTEGER DIVISION: selection center = (min+max)/2 with C++ truncation; in Lua use math.floor((min+max)/2) (coords are non-negative so floor == trunc). Same for any /100 integer math; densitySpin/100.0f must be float division. 2. SORTED-MAP ITERATION ORDER IS LOAD-BEARING in BuildRule: clusterTiles must be emitted sorted by (offset.x asc, then offset.y asc), and ItemEntries sorted by itemId asc; Lua tables have no order — collect keys and table.sort with an explicit comparator replicating std::pair/uint16 ordering.
