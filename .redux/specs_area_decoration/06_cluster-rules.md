## Summary
Cluster rules treat rule.clusterTiles as a SEARCH TEMPLATE: the engine scans every area tile, finds all positions where the template's item pattern exists exactly on the map (sub-multiset match per tile, all footprint tiles inside the area), shuffles those candidates with the shared mt19937, then tries to place up to rule.instanceCount instances enforcing instanceMinDistance via an INLINE Chebyshev check against a local list (NOT checkClusterCenterSpacing, despite that method existing for other code paths). Centered mode (rule.hasCenterPoint==true) places one weighted-random item from rule.items at each matched center, with spacing checks; Random mode (hasCenterPoint==false) anchors on clusterTiles[0] and scatters items over every template tile with a per-tile density roll — and performs NO spacing check. requireGround is NOT all-or-nothing: it is a per-composite-tile skip inside buildPlacementItems, and is never consulted for simple (non-composite) item entries. rule.maxPlacements and friend-floor bias are completely ignored in cluster mode; preset.maxItemsTotal is enforced.

## RNG
All draws come from the single shared std::mt19937 m_rng, seeded earlier with static_cast<unsigned int>(seed) (32-bit truncation of the 64-bit seed). Cluster rules run AFTER the distribution pass, so the RNG stream position depends on everything that pass consumed. Per cluster rule, in execution order:

generateClusterCentered:
 1. std::shuffle(candidatePositions, m_rng) — Fisher-Yates; the number/values of underlying mt19937 invocations are IMPLEMENTATION-DEFINED (MSVC vs libstdc++ differ). Consumed even if instanceCount <= 0.
 2. Per candidate that survives the instanceMinDistance check: selectItemFromRule — exactly ONE uniform_int_distribution<int>(0, totalWeight-1) draw (zero draws if totalWeight <= 0). Drawn even for candidates later rejected by validate/build/cap/spacing — rejected candidates still advance the stream.
 3. If the selected ItemEntry has isCluster==true: buildPlacementItems draws pairs (dx, dy) from uniform_int_distribution<int>(-radius, radius), 2 draws per attempt, up to 20 attempts per additional center, for centers i = 1 .. max(1,clusterCount)-1. dx is drawn before dy. Non-cluster entries draw nothing here.

generateClusterRandom:
 1. std::shuffle(candidatePositions, m_rng) — same caveats.
 2. Per candidate that survives the instanceMinDistance check, iterating rule.clusterTiles in order, PER template tile: (a) one uniform_real_distribution<float>(0.0f, 1.0f) density draw — ALWAYS drawn first; if draw > rule.density the tile is skipped and NO further draws happen for it; (b) otherwise one selectItemFromRule int draw; (c) if that entry is an ItemEntry-cluster, the (dx,dy) scatter draws as above. Failures at validate/build do not roll back draws.

Determinism: exact sequence matters everywhere — the same seed only reproduces the same map if every draw (shuffle internals, uniform_int, uniform_real<float>) is reproduced bit-for-bit. std::shuffle, uniform_int_distribution, and uniform_real_distribution algorithms are all implementation-defined in C++; a Lua port CANNOT match C++ output across stdlibs unless it reimplements one specific stdlib's algorithms (MSVC here, since the project builds on Windows). For a self-consistent Lua port, define: mt19937 core + a fixed Lemire/modulo int-bounding + ldexp-based float-in-[0,1) + explicit Fisher-Yates (i from n-1 down to 1, j = uniform_int(0, i)), and keep the draw ORDER exactly as listed.

Likely bugs to preserve as-is (flagging): (1) generateClusterRandom never calls checkSpacingForPlacement — SpacingConfig is unenforced for random-mode cluster placements while centered mode enforces it; (2) the per-placement maxItemsTotal overflow check uses continue and never sets m_previewWasCapped, so capping can occur unreported; (3) rule.maxPlacements is ignored in both cluster generators; (4) RNG draws happen for candidates that are subsequently rejected, so adding/removing a rejection path changes downstream randomness; (5) zero-radius ItemEntry clusters burn 40 RNG draws achieving nothing (see edge case 9).

Lua pitfalls: (a) 1-based arrays — clusterTiles[0] anchor in C++ is clusterTiles[1] in Lua; selectItemFromRule's cumulative walk and "first nonzero id" scans must keep C++ vector order; (b) iteration order: candidate enumeration follows the `tiles` vector order (deterministic), clusterTiles and rule.items follow their definition/serialization order, instanceCenters/instanceAnchors follow insertion order — use array tables with ipairs, NEVER pairs over hash tables, including for the validPositions set (set membership only, fine as hash keys, but never iterate it); (e) uniform_real_distribution<float>(0,1) yields [0,1) floats — the density comparison `draw > rule.density` with density stored as float 1.0 always passes; in Lua use the same half-open range; beware float32 vs float64: C++ compares float against float, Lua compares doubles — for densities strictly between 0 and 1 the practical difference is negligible but not bit-identical; (f) std::shuffle / uniform_int / uniform_real are stdlib-implementation-defined: cross-language seed-compatibility with the C++ build is effectively impossible without reimplementing MSVC's exact algorithms; recommend defining the Lua port's own canonical mt19937 + Fisher-Yates(i=n-1..1, j=rand(0..i)) + bounded-int + float-in-[0,1) and documenting that seeds are port-internal, while preserving DRAW ORDER exactly as specified in rngUsage so the port is self-deterministic;
